# PROJECT-TASK MANAGEMENT API
In this api, we'll have projects that have tasks. We'll follow DDD

- ## UUID in domain models
We have a bit of a problem: the ids, autogenerated by the DB are an issue for our equals and hashcode methods which we need to have for our Sets. We use an UUID to uniquely identify an entity from the domain perspective which helps the identification for our Sets. 

- ## Not using the unique name for equality
Name is something, that, even though is unique, can change. What this means for our project is that, at any point, a saved project's hashcode can change if the name is changed. 

- ## addTaskEntity method in ProjectEntity
We do this to keep both sides of the relationship updated. This, because the TaskEntity is the owner (relational DBs can't store collections) so, to handle the relationship, there's a column with the FK of the parent in the child entity. So what we think of as a Set, is actually a table with a lot of tasks that have the id of the project they belong to.
